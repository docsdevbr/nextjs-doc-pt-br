### Arquivos de roteamento

Adicione `page` para expor uma rota, `layout` para interfaces da pessoa usuária
compartilhadas, como cabeçalho, navegação ou rodapé, `loading` para esqueletos
de código, `error` para fronteiras de erro e `route` para APIs.

|                                                                               |                     |                                         |
| ----------------------------------------------------------------------------- | ------------------- |-----------------------------------------|
| [`layout`](/docs/app/api-reference/file-conventions/layout)                   | `.js` `.jsx` `.tsx` | Layout                                  |
| [`page`](/docs/app/api-reference/file-conventions/page)                       | `.js` `.jsx` `.tsx` | Página                                  |
| [`loading`](/docs/app/api-reference/file-conventions/loading)                 | `.js` `.jsx` `.tsx` | UI de carregamento                      |
| [`not-found`](/docs/app/api-reference/file-conventions/not-found)             | `.js` `.jsx` `.tsx` | UI de arquivo não encontrado            |
| [`error`](/docs/app/api-reference/file-conventions/error)                     | `.js` `.jsx` `.tsx` | UI de erro                              |
| [`global-error`](/docs/app/api-reference/file-conventions/error#global-error) | `.js` `.jsx` `.tsx` | UI de erro global                       |
| [`route`](/docs/app/api-reference/file-conventions/route)                     | `.js` `.ts`         | Endpoint de API                         |

| [`default`](/docs/app/api-reference/file-conventions/default)                 | `.js` `.jsx` `.tsx` | Página de fallback para rotas paralelas |

### Rotas aninhadas

Pastas definem segmentos de URL.
Pastas aninhadas aninham segmentos.
Layouts em qualquer nível envolvem seus segmentos filhos.
Uma rota se torna pública quando um arquivo `page` ou `route` existe.

| Caminho                     | Padrão de URL   | Observações                         |
|-----------------------------|-----------------|-------------------------------------|
| `app/layout.tsx`            | —               | Layout raiz envolve todas as rotas  |
| `app/blog/layout.tsx`       | —               | Envolve `/blog` e seus descendentes |
| `app/page.tsx`              | `/`             | Rota pública                        |
| `app/blog/page.tsx`         | `/blog`         | Rota pública                        |
| `app/blog/authors/page.tsx` | `/blog/authors` | Rota pública                        |

### Rotas dinâmicas

Parametrize segmentos com colchetes.
Use `[segmento]` para um único parâmetro, `[...segmento]` para um parâmetro
genérico e `[[...segmento]]` para um parâmetro genérico opcional.
Acesse os valores através da propriedade
`[`params`](/docs/app/api-reference/file-conventions/page#params-optional)`.

| Caminho                         | Padrão de URL                                                        |
|---------------------------------|----------------------------------------------------------------------|
| `app/blog/[slug]/page.tsx`      | `/blog/my-first-post`                                                |
| `app/shop/[...slug]/page.tsx`   | `/shop/clothing`, `/shop/clothing/shirts`                            |
| `app/docs/[[...slug]]/page.tsx` | `/docs`, `/docs/layouts-and-pages`, `/docs/api-reference/use-router` |

### Grupos de rotas e pastas privadas

Organize o código sem alterar URLs com grupos de rotas
[`(grupo)`](/docs/app/api-reference/file-conventions/route-groups#convention) e
agrupe arquivos não roteáveis em pastas privadas
[`_pasta`](#pastas-privadas).

| Caminho                         | Padrão de URL | Observações                                                                |
|---------------------------------|---------------|----------------------------------------------------------------------------|
| `app/(marketing)/page.tsx`      | `/`           | Grupo omitido da URL                                                       |
| `app/(shop)/cart/page.tsx`      | `/cart`       | Compartilha layouts dentro de `(shop)`                                     |
| `app/blog/_components/Post.tsx` | —             | Não roteável; local seguro para utilitários de interface da pessoa usuária |
| `app/blog/_lib/data.ts`         | —             | Não roteável; local seguro para utilitários                                |

### Rotas paralelas e interceptadas

Esses recursos se encaixam em padrões específicos de interface da pessoa
usuária, como layouts baseados em slots ou roteamento modal.

Use `@slot` para slots nomeados renderizados por um layout pai.
Use padrões de interceptação para renderizar outra rota dentro do layout atual
sem alterar a URL, por exemplo, para exibir uma visualização de detalhes como um
modal sobre uma lista.

| Padrão (docs)                                                                               | Significado                 | Caso de uso típico                            |
|---------------------------------------------------------------------------------------------|-----------------------------|-----------------------------------------------|
| [`@folder`](/docs/app/api-reference/file-conventions/parallel-routes#slots)                 | Slot nomeado                | Barra lateral + conteúdo principal            |
| [`(.)folder`](/docs/app/api-reference/file-conventions/intercepting-routes#convention)      | Intercepta no mesmo nível   | Visualizar rota irmã em um modal              |
| [`(..)folder`](/docs/app/api-reference/file-conventions/intercepting-routes#convention)     | Intercepta o pai            | Abrir um filho do pai como uma sobreposição   |
| [`(..)(..)folder`](/docs/app/api-reference/file-conventions/intercepting-routes#convention) | Intercepta dois níveis      | Sobreposição profundamente aninhada           |
| [`(...)folder`](/docs/app/api-reference/file-conventions/intercepting-routes#convention)    | Intercepta a partir da raiz | Mostrar rota arbitrária na visualização atual |

## Organizando seu projeto

O Next.js **não impõe** uma forma específica de organizar e alocar os arquivos
do seu projeto.
No entanto, ele oferece diversos recursos para ajudar na organização.

### Hierarquia de componentes

Os componentes definidos em arquivos específicos são renderizados em uma
hierarquia específica:

- `layout.js`
- `template.js`
- `error.js` (fronteira de erro do React)
- `loading.js` (fronteira de suspense do React)
- `not-found.js` (fronteira de erro do React para a UI "não encontrado")
- `page.js` ou `layout.js` aninhado

<Image
  alt="Hierarquia de componentes para convenções de arquivos"
  srcLight="/docs/light/file-conventions-component-hierarchy.png"
  srcDark="/docs/dark/file-conventions-component-hierarchy.png"
  width="1600"
  height="643"
/>

Os componentes são renderizados recursivamente em rotas aninhadas, o que
significa que os componentes de um segmento de rota serão aninhados **dentro**
dos componentes de seu segmento pai.

<Image
  alt="Hierarquia de componentes das convenções de arquivos aninhados"
  srcLight="/docs/light/nested-file-conventions-component-hierarchy.png"
  srcDark="/docs/dark/nested-file-conventions-component-hierarchy.png"
  width="1600"
  height="863"
/>

### Colocação

No diretório `app`, pastas aninhadas definem a estrutura de rotas.
Cada pasta representa um segmento de rota mapeado para um segmento
correspondente em um caminho de URL.

No entanto, mesmo que a estrutura de rotas seja definida por meio de pastas, uma
rota **não é publicamente acessível** até que um arquivo `page.js` ou `route.js`
seja adicionado a um segmento de rota.

<Image
  alt="Um diagrama mostrando como uma rota não é publicamente acessível até que um arquivo `page.js` ou `route.js` seja adicionado a um segmento de rota."
  srcLight="/docs/light/project-organization-not-routable.png"
  srcDark="/docs/dark/project-organization-not-routable.png"
  width="1600"
  height="444"
/>

E, mesmo quando uma rota é tornada publicamente acessível, apenas o **conteúdo
retornado** por `page.js` ou `route.js` é enviado ao cliente.

<Image
  alt="Um diagrama mostrando como os arquivos page.js e route.js tornam as rotas publicamente acessíveis."
  srcLight="/docs/light/project-organization-routable.png"
  srcDark="/docs/dark/project-organization-routable.png"
  width="1600"
  height="687"
/>

Isso significa que os **arquivos de projeto** podem ser **colocados em conjunto
com segurança** dentro de segmentos de rota no diretório `app` sem se tornarem
roteáveis acidentalmente.

<Image
  alt="Um diagrama mostrando que arquivos de projeto colocados em conjunto não são roteáveis, mesmo quando um segmento contém um arquivo page.js ou route.js."
  srcLight="/docs/light/project-organization-colocation.png"
  srcDark="/docs/dark/project-organization-colocation.png"
  width="1600"
  height="1011"
/>

> **Bom saber**: Embora você **possa** colocar os arquivos do seu projeto no
> diretório `app`, você não **precisa** fazer isso.
> Se preferir, você pode
> [mantê-los fora do diretório `app`](#armazene-arquivos-do-projeto-fora-de-app).

### Pastas privadas

Pastas privadas podem ser criadas prefixando o nome da pasta com um sublinhado:
`_nomeDaPasta`.

Isso indica que a pasta é um detalhe de implementação privado e não deve ser
considerada pelo sistema de roteamento, **excluindo a pasta e todas as suas
subpastas** do roteamento.

<Image
  alt="Um exemplo de estrutura de pastas usando pastas privadas"
  srcLight="/docs/light/project-organization-private-folders.png"
  srcDark="/docs/dark/project-organization-private-folders.png"
  width="1600"
  height="849"
/>

Como os arquivos no diretório `app` podem ser
[colocados com segurança por padrão](#colocação), pastas privadas não são
obrigatórias para a colocalização.
No entanto, elas podem ser úteis para:

- Separar a lógica da UI da lógica de roteamento.
- Organizar arquivos internos de forma consistente em todo o projeto e no
  ecossistema Next.js.
- Ordenar e agrupar arquivos em editores de código.
- Evitar possíveis conflitos de nomenclatura com futuras convenções de arquivos
  do Next.js.

> **Bom saber**:
>
> - Embora não seja uma convenção do framework, você também pode considerar
>   marcar arquivos fora de pastas privadas como "privados" usando o mesmo
>   padrão de sublinhado.
> - Você pode criar segmentos de URL que começam com um sublinhado prefixando o
>   nome da pasta com `%5F` (a forma codificada em URL de um sublinhado):
>   `%5FfolderName`.
> - Se você não usa pastas privadas, é útil conhecer as
>   [convenções de arquivos especiais](/docs/app/getting-started/project-structure#arquivos-de-roteamento)
>   do Next.js para evitar conflitos de nomes inesperados.

### Grupos de rotas

Grupos de rotas podem ser criados envolvendo uma pasta entre parênteses:
`(nomeDaPasta)`

Isso indica que a pasta é para fins de organização e **não deve ser incluída**
no caminho da URL da rota.

<Image
  alt="Um exemplo de estrutura de pastas usando grupos de rotas"
  srcLight="/docs/light/project-organization-route-groups.png"
  srcDark="/docs/dark/project-organization-route-groups.png"
  width="1600"
  height="849"
/>

Os grupos de rotas são úteis para:

- Organizar rotas por seção do site, intenção ou time.
  Ex.: páginas de marketing, páginas administrativas, etc.
- Habilitar layouts aninhados no mesmo nível de segmento de rota:
  - [Criando vários layouts aninhados no mesmo segmento, incluindo vários layouts raiz](#criando-vários-layouts-raiz)
  - [Adicionando um layout a um subconjunto de rotas em um segmento comum](#adicionando-segmentos-específicos-a-um-layout)

### Pasta `src`

O Next.js permite armazenar o código da aplicação (incluindo `app`) em uma
[pasta `src`](/docs/app/api-reference/file-conventions/src-folder) opcional.
Isso separa o código da aplicação dos arquivos de configuração do projeto, que
geralmente ficam na raiz do projeto.

## Exemplos

A seção a seguir apresenta uma visão geral de alto nível das estratégias mais
comuns.
A principal conclusão é escolher uma estratégia que funcione para você e seu
time e manter a consistência em todo o projeto.

> **Bom saber**: Nos exemplos abaixo, usamos as pastas `components` e `lib` como
> marcadores genéricos.
> Seus nomes não têm significado específico para nenhum framework e seus
> projetos podem usar outras pastas, como `ui`, `utils`, `hooks`, `styles`, etc.

### Armazene arquivos do projeto fora de `app`

Essa estratégia armazena todo o código da aplicação em pastas compartilhadas na
**raiz do seu projeto** e mantém o diretório `app` apenas para fins de
roteamento.

<Image
  alt="Exemplo de estrutura de pastas com arquivos do projeto fora de app"

  srcDark="/docs/dark/project-organization-project-root.png"
  width="1600"
  height="849"
/>

### Armazene os arquivos do projeto em pastas de nível superior dentro de `app`

Essa estratégia armazena todo o código da aplicação em pastas compartilhadas na
**raiz do diretório `app`**.

<Image
  alt="Exemplo de estrutura de pastas com arquivos de projeto dentro de app"
  srcLight="/docs/light/project-organization-app-root.png"
  srcDark="/docs/dark/project-organization-app-root.png"

### Separe os arquivos do projeto por funcionalidade ou rota

Essa estratégia armazena o código da aplicação compartilhado globalmente no
diretório raiz `app` e **separa** o código mais específico da aplicação nos
segmentos de rota que o utilizam.

<Image

### Organize as rotas sem afetar o caminho da URL

Para organizar as rotas sem afetar a URL, crie um grupo para manter rotas
relacionadas juntas.
As pastas entre parênteses serão omitidas da URL (por exemplo, `(marketing)` ou
`(shop)`).

<Image
  alt="Organizando rotas com grupos de rotas"
  srcLight="/docs/light/route-group-organisation.png"
  srcDark="/docs/dark/route-group-organisation.png"
  width="1600"

### Adicionando segmentos específicos a um layout

Para adicionar rotas específicas a um layout, crie um novo grupo de rotas (por
exemplo, `(shop)`) e mova as rotas que compartilham o mesmo layout para dentro
do grupo (por exemplo, `account` e `cart`).
As rotas fora do grupo não compartilharão o layout (por exemplo, `checkout`).

### Adicionando esqueletos de carregamento em uma rota específica

Para aplicar um
[esqueleto de carregamento](/docs/app/api-reference/file-conventions/loading)
por meio de um arquivo `loading.js` a uma rota específica, crie um novo grupo de
rotas (por exemplo, `/(overview)`) e mova seu arquivo `loading.tsx` para dentro
desse grupo de rotas.

<Image
  alt="Estrutura de pastas mostrando um arquivo loading.tsx e um arquivo page.tsx dentro do grupo de rotas"
  srcLight="/docs/light/route-group-loading.png"
  srcDark="/docs/dark/route-group-loading.png"
  width="1600"
  height="444"
/>

Agora, o arquivo `loading.tsx` será aplicado somente à sua página dashboard →
overview, e não a todas as páginas do dashboard, sem afetar a estrutura do
caminho da URL.

### Criando vários layouts raiz

Para criar vários
[layouts raiz](/docs/app/api-reference/file-conventions/layout#root-layout),
remova o arquivo `layout.js` de nível superior e adicione um arquivo `layout.js`
dentro de cada grupo de rotas.
Isso é útil para dividir uma aplicação em seções com UI ou experiências
completamente diferentes.
As tags `<html>` e `<body>` precisam ser adicionadas a cada layout raiz.

No exemplo acima, tanto `(marketing)` quanto `(shop)` têm seu próprio layout
raiz.
